# Plan: Federated Learning System for Rottra

## Overview

Triển khai **Federated Learning (FL)** cho Rottra — cho phép nhiều farm/node huấn luyện model cục bộ và chia sẻ updates mà không chia sẻ dữ liệu thô. Hệ thống tích hợp với infrastructure AI hiện có.

## Problem Statement

- Nông dân muốn đóng góp dữ liệu để cải thiện AI chung nhưng không muốn lộ dữ liệu riêng tư
- Mỗi farm có điều kiện khác nhau (đất, khí hậu, mùa vụ) — model cần học từ đa dạng dữ liệu
- Cần cơ chế verifiable để đảm bảo training integrity

## Core Components

### 1. Federated Learning Coordinator (`src/core/federated-learning/coordinator.ts`)

Central orchestrator managing FL rounds:

```typescript
class FederatedLearningCoordinator {
  // Start a new FL round
  async startRound(config: FLRoundConfig): Promise<FLRound>
  
  // Aggregate gradients from participating nodes
  async aggregateGradients(gradients: GradientUpdate[]): Promise<AggregatedModel>
  
  // Broadcast updated global model
  async broadcastModel(roundId: string): Promise<void>
  
  // Track round lifecycle
  async getRoundStatus(roundId: string): Promise<FLRoundStatus>
}
```

**FedAvg Algorithm Implementation:**
- Weighted averaging based on node data size
- Differential privacy noise injection
- Secure aggregation (no single gradient exposed)

### 2. Local Training Engine (`src/core/federated-learning/local-trainer.ts`)

Runs on each farm node:

```typescript
class LocalTrainer {
```

## Database Schema

### New Tables

```sql
-- FL Round tracking
CREATE TABLE fl_round (
  id TEXT PRIMARY KEY,
  round_number INTEGER NOT NULL,
  status VARCHAR(50) DEFAULT 'pending', -- pending/aggregating/completed/failed
  global_model_id TEXT,
  config JSONB,
  started_at TIMESTAMP,
  completed_at TIMESTAMP,
  participant_count INTEGER,
  aggregation_method VARCHAR(50) DEFAULT 'fedavg'
);

-- Gradient updates from nodes
CREATE TABLE fl_gradient_update (
  id TEXT PRIMARY KEY,
  round_id TEXT REFERENCES fl_round(id),
  node_id TEXT NOT NULL,
  model_version VARCHAR(100),
  gradient_hash VARCHAR(256), -- SHA-256 of gradients
  data_sample_count INTEGER,
  local_loss REAL,
  local_accuracy REAL,
  encrypted_gradients JSONB, -- Encrypted gradient data
  submitted_at TIMESTAMP,
  status VARCHAR(50) DEFAULT 'pending'
);

-- Model versions
CREATE TABLE fl_model_version (
  id TEXT PRIMARY KEY,
  version_number INTEGER NOT NULL,
  round_id TEXT REFERENCES fl_round(id),
  model_weights JSONB, -- Serialized model weights
  model_hash VARCHAR(256), -- SHA-256 of model
  parent_version_id TEXT,
  metrics JSONB, -- accuracy, loss, etc.
  created_at TIMESTAMP
);

-- Node participation
CREATE TABLE fl_node (
  id TEXT PRIMARY KEY,
  node_name VARCHAR(255),
  farm_id TEXT REFERENCES farm(id),
  public_key TEXT, -- For secure aggregation
  last_seen TIMESTAMP,
  total_rounds_participated INTEGER DEFAULT 0,
  reputation_score REAL DEFAULT 1.0
);

-- Privacy budget tracking
CREATE TABLE fl_privacy_budget (
  id TEXT PRIMARY KEY,
  node_id TEXT REFERENCES fl_node(id),
  epsilon_used REAL DEFAULT 0,
  epsilon_limit REAL DEFAULT 10.0,
  delta_used REAL DEFAULT 0,
  round_count INTEGER DEFAULT 0,
  updated_at TIMESTAMP
);
```

## API Endpoints

### Coordinator APIs

```typescript
// Start new FL round
POST /api/fl/start-round
Body: { config: FLRoundConfig }

// Get FL status
GET /api/fl/status
Response: { activeRounds: FLRound[], nodes: FLNode[] }
```

```typescript

// Report local training metrics
POST /api/fl/local-metrics
```

## Key Design Decisions

### 1. FedAvg vs FedProx
- **Decision**: FedAvg (simpler, proven)
- **Rationale**: Agricultural data is relatively homogeneous; FedProx needed for heterogeneous data

### 2. Centralized vs Decentralized
- **Decision**: Centralized coordinator (existing PostgreSQL)
- **Rationale**: Rottra already has central DB; decentralized would require P2P protocol

### 3. Privacy Budget
- **Decision**: ε = 1.0 per round, δ = 1e-5
- **Rationale**: Balance between privacy and model utility for agricultural data

### 4. Minimum Participation
- **Decision**: 3 nodes minimum for aggregation
- **Rationale**: Prevent model poisoning from single malicious node

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Gradient leakage | DP noise + secure aggregation |
| Model poisoning | Minimum participation + Byzantine fault tolerance |
| Communication overhead | Async gradient exchange + compression |
| Node failures | Round timeout + retry mechanism |

## Success Metrics

- **Privacy**: ε ≤ 1.0 per round
- **Model Quality**: ≥ 90% of centralized training accuracy
- **Communication**: < 10MB per round per node
- **Scalability**: Support 10+ concurrent nodes
